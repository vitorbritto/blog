---
title: Evaluating context files
excerpt: Are repository-level context files useful for coding agents?

---

A fairly common practice in software development is adapting coding agents to specific repositories through context files, such as manually written or automatically generated `AGENTS.md` files.

Although this practice is strongly recommended by agent developers, there had not yet been a rigorous investigation into whether these files actually help with real tasks.

To better understand this scenario, I read the paper "Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?" written by Thibaud Gloaguen, Niels Mündler, Mark Müller, Veselin Raychev, and Martin Vechev, published on February 12, 2026.

The paper has 18 pages. In this work, the authors study this question and evaluate the performance of coding agents in two complementary scenarios:

1. established SWE-bench* tasks from popular repositories, using context files generated by LLMs following recommendations from agent developers themselves;
2. a new collection of _issues_ from repositories that already had context files written by their own developers.

> * SWE-bench is a benchmark for evaluating coding agents on problem-solving tasks. It includes a variety of tasks, such as bug fixes, feature implementation, and code refactoring, all extracted from real GitHub repositories.

Across several agents and language models, the authors found that context files tend to reduce task success rates compared with providing no repository context at all, while also increasing inference cost by more than 20%.

From a behavioral perspective, both LLM-generated files and developer-written files encourage broader repository exploration, such as more tests and more file navigation. Agents also tend to follow the instructions in these files.

The main conclusion is that unnecessary requirements placed in these files make tasks harder. Because of that, human-written files should contain only the minimum requirements that are truly necessary.

## Main idea of the article

The paper questions a practice that has become almost standard when using coding agents: creating files such as `AGENTS.md`, `CLAUDE.md`, or similar files to guide tools like Codex, Claude Code, Qwen Code, and other agents.

The "popular" hypothesis is: "If I give the agent more context about the repository, it will perform better."

However, the paper shows that this is not always true.

In practice, more context can make the agent:

* explore more files;
* run more tests;
* follow more instructions;
* spend more tokens;
* take longer;
* and, in some cases, solve fewer tasks.

In other words: too much context can become operational noise.

## Key findings

### 1. LLM-generated files tend to slightly hurt performance

Automatically generated context files had a small negative effect on agent success rates.

On [SWE-bench Lite](https://www.swebench.com/lite.html), the average drop was small. On [AGENTBENCH](https://github.com/THUDM/AgentBench), the benchmark created by the authors, the average drop was also small, but consistent enough to question the usefulness of these automatically generated files.

### 2. Human-written files help a little, but increase cost

Files created by developers performed better than LLM-generated files. Even so, the gain was modest.

They improved average performance by around 4%, but also increased the number of steps and the execution cost.

### 3. Agents follow file instructions

> **An important point:** the problem is not that agents ignore `[AGENTS|CLAUDE].md`.
> Quite the opposite. They follow it.

When the file mentions tools such as `uv`, `pytest`, or repository-specific tools, agents start using them more.

That is exactly the problem: if the file has too many instructions, the agent can spend energy obeying things that do not directly help solve the task.

### 4. Context files do not work that well as a "repository map"

A common recommendation is to use `AGENTS.md` to provide an overview of the repository.

But the study shows that this did not help agents find the relevant files for fixing an issue more quickly.

In some cases, agents even spent more steps reading, rereading, and searching for these context files, even when they were already present in the initial context.

### 5. Context can be useful when the repository has little documentation

One interesting finding: when the authors removed documentation files from the repository, such as `.md` files, examples, and the `docs/` folder, context files became more helpful.

This suggests that `AGENTS.md` can be useful in poorly documented repositories. However, in repositories that already have good documentation, it may end up being redundant.

## Conclusion

The study shows that repository-level context files are not a magic solution for improving the performance of coding agents. They change agent behavior, but that change does not always translate into better results.

In practice, agents read and follow these instructions. The problem is that following more instructions also means taking more steps, exploring more files, running more tests, spending more tokens, and, in some cases, moving away from the most direct path to solving the task.

LLM-generated files seem especially fragile because they tend to turn a superficial view of the repository into rules that the agent then obeys. Developer-written files can help, but the observed gain is small and comes with a higher operational cost.

The main lesson from the paper is that context needs to be treated as a precision tool, not as an information dump. A good `AGENTS.md`, `CLAUDE.md`, or equivalent file should not try to explain the entire repository. It should record only what actually changes task execution:

- important test commands;
- conventions that must not be broken;
- architectural decisions that are still valid;
- known project pitfalls;
- constraints the agent would be unlikely to discover on its own.

Therefore, the right question is not "how can I give the agent more context?", but "which context is worth the cost of being in the agent's path?".
