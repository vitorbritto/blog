Se você é desenvolvedor, provavelmente já repetiu os "quatro pilares" da Orientação a Objetos como se eles fossem a fundação definitiva do paradigma: abstração, encapsulamento, herança e polimorfismo.

Eles são úteis como introdução didática. O problema começa quando essa lista vira dogma. Muitos sistemas escritos "seguindo os pilares" ainda acabam rígidos, difíceis de depurar e sensíveis a mudanças pequenas demais para causar tanto estrago.

Então a pergunta real não é se abstração, encapsulamento, herança e polimorfismo existem. Eles existem. A pergunta é outra: eles capturam mesmo o coração da orientação a objetos?

No sentido defendido por Alan Kay, a resposta é: não completamente.

## A troca de mensagens

Alan Kay, que cunhou o termo object-oriented programming e liderou a criação das primeiras versões de Smalltalk, descreveu OO de uma forma bem diferente da lista popular dos quatro pilares. Para ele, OO envolve mensagens, retenção local, proteção do estado-processo e late binding extremo.

**Ou seja**, o centro da ideia não é "classe", nem "herança", nem "objeto como estrutura de dados com funções acopladas". O centro é um conjunto de entidades autônomas que preservam seu próprio estado e interagem por mensagens.

> A essência não são apenas as "caixas". São também as "setas".

Uma forma útil de visualizar isso aparece em apresentações sobre Alan Kay por meio do conceito japonês de Ma: o espaço entre as coisas. A metáfora ajuda porque desloca a atenção do interior do objeto para o protocolo de interação entre objetos. O ponto não é endeusar uma palavra japonesa, mas lembrar que OO nasceu muito mais próxima de redes de "agentes comunicantes" do que de árvores de classes.

Isso também exige uma distinção técnica importante. Ao falar em "chamar um método", normalmente descrevemos uma operação direta sobre uma implementação conhecida. Já "enviar uma mensagem", no sentido de Smalltalk e Objective-C, implica um modelo em que a mensagem é despachada para um receptor, e a resolução do comportamento pode acontecer tardiamente.

Em Smalltalk, quando um objeto recebe uma mensagem que não entende, o fluxo passa por mecanismos como `doesNotUnderstand:`. Em Objective-C, há mecanismos como _dynamic method resolution_ e _message forwarding_. Isso permite que o receptor participe da decisão sobre como lidar com a mensagem em tempo de execução.

Sem esse grau de flexibilidade, uma linguagem ainda pode ser chamada de orientada a objetos no uso comum da indústria de Software. Porém, ela se afasta do sentido original enfatizado por Kay e se aproxima mais do ramo class-based/ADT: tipos, operações e hierarquias, com menos autonomia dinâmica entre objetos.

## A abstração sem contexto empobrece o domínio

Fomos ensinados que abstrair é sempre uma virtude. Richard Gabriel e James Coplien são bem mais cuidadosos que isso. A crítica deles não é contra toda abstração, mas contra a abstração excessiva, genérica demais ou desconectada do contexto humano e do domínio.

Gabriel, em ["Abstraction Descant"](https://www.dreamsongs.com/Files/PatternsOfSoftware.pdf), argumenta que abstrações podem tornar o software menos habitável quando escondem detalhes que o programador precisa entender para manter o sistema. Coplien, no prefácio de *Clean Code*, resume essa crítica de forma provocativa ao dizer que Gabriel nos aconselha que abstração é "má". A frase é forte, mas não deve ser lida como rejeição total à abstração. O alvo é a abstração que joga fora informação relevante.

Uma abstração saudável reduz carga cognitiva sem apagar o significado. Uma abstração ruim troca linguagem de domínio por nomes genéricos como `Manager`, `Processor`, `Handler`, `Validator` e `Engine`, que parecem flexíveis, mas frequentemente escondem o que o sistema realmente faz.

Uma alternativa é pensar em compressão semântica. Aqui, uso "compressão" como uma aplicação prática inspirada por Gabriel: não no sentido de criar uma camada genérica, mas de condensar detalhes operacionais em um conceito que preserve significado.

Em vez de criar um `RuleEngine`, `Validator`, `Processor` ou `AbstractCreditPolicy`, podemos criar um conceito de domínio: `CreditProfile`.

Veja este exemplo simples de um processo de decisão de crédito escrito em Go:

```go
package main

import "fmt"

type Money int64 // cents

type CreditApplication struct {
	CustomerID           string
	MonthlyIncome        Money
	CurrentMonthlyDebt   Money
	RequestedInstallment Money
	EmploymentMonths     int
	HasRecentDefault     bool
}

type CreditProfile struct {
	DebtBurdenPercent int
	IncomeIsStable    bool
	HasRecentDefault  bool
}

type CreditDecision struct {
	Approved bool
	Reason   string
}

func BuildCreditProfile(app CreditApplication) CreditProfile {
	totalDebtAfterLoan := app.CurrentMonthlyDebt + app.RequestedInstallment

	return CreditProfile{
		DebtBurdenPercent: percentage(totalDebtAfterLoan, app.MonthlyIncome),
		IncomeIsStable:    app.EmploymentMonths >= 12,
		HasRecentDefault:  app.HasRecentDefault,
	}
}

func DecideCredit(profile CreditProfile) CreditDecision {
	if profile.HasRecentDefault {
		return CreditDecision{
			Approved: false,
			Reason:   "customer has a recent default",
		}
	}

	if !profile.IncomeIsStable {
		return CreditDecision{
			Approved: false,
			Reason:   "income is not stable enough",
		}
	}

	if profile.DebtBurdenPercent > 35 {
		return CreditDecision{
			Approved: false,
			Reason:   "debt burden would exceed 35% of monthly income",
		}
	}

	return CreditDecision{
		Approved: true,
		Reason:   "credit profile is within policy",
	}
}

func percentage(value Money, total Money) int {
	if total == 0 {
		return 100
	}

	return int(value * 100 / total)
}

func main() {
	app := CreditApplication{
		CustomerID:           "customer_id",
		MonthlyIncome:        500_000,
		CurrentMonthlyDebt:   90_000,
		RequestedInstallment: 80_000,
		EmploymentMonths:     18,
		HasRecentDefault:     false,
	}

	creditProfile := BuildCreditProfile(app)
	creditDecision := DecideCredit(creditProfile)

	fmt.Printf("Debt burden: %d%%\n", creditProfile.DebtBurdenPercent)
	fmt.Printf("Approved: %t\n", creditDecision.Approved)
	fmt.Printf("Reason: %s\n", creditDecision.Reason)
}
```

O ponto aqui não é dizer que `CreditProfile` deixa de ser uma abstração. Muito pelo contrário, ele é uma abstração. O ponto central é que ele é uma abstração ancorada no domínio.

Antes, alguém precisaria reconstruir mentalmente o significado de:

```go
app.CurrentMonthlyDebt + app.RequestedInstallment
app.MonthlyIncome
app.EmploymentMonths >= 12
app.HasRecentDefault
```

Depois, o sistema passa a se comunicar por meio de um conceito de domínio que já carrega significado:

```go
creditProfile.DebtBurdenPercent
creditProfile.IncomeIsStable
creditProfile.HasRecentDefault
```

Isso reduz a carga cognitiva sem apagar a realidade. O código continua próximo do domínio, mas passa a comunicar intenção de forma mais direta.

## Sistemas vivem nas interações

Russell Ackoff não escreveu sobre *microservices*, *packages* ou *code coupling*. A contribuição dele vem de outro lugar: pensamento sistêmico.

> Se você não assistiu a [palestra de Ackoff sobre pensamento sistêmico](https://www.youtube.com/watch?v=O9TE9HWFo6U), pare tudo e assista agora mesmo. O vídeo é antigo, mas a mensagem é atemporal. Ele explica de forma clara e acessível que um sistema é definido por suas interações, não por suas partes isoladas.

Para Ackoff, as propriedades essenciais de um sistema não surgem das partes isoladas, mas das interações entre elas. Um carro desmontado pode conter todas as peças, mas já não tem a propriedade de transportar alguém. A propriedade aparece no todo organizado.

Aplicando essa ideia ao software, o problema não é simplesmente "ter acoplamento". O problema é ter dependências que não correspondem ao domínio, ou dependências reais escondidas atrás de uma arquitetura que finge independência.

Desacoplamento cego pode destruir visibilidade. Em sistemas distribuídos mal desenhados, as dependências continuam existindo, mas passam a morar em filas, payloads, contratos implícitos, convenções de banco, *retries*, *dashboards* e *tribal knowledge*. O sistema parece modular no diagrama, mas se comporta como uma rede opaca em produção.

Isso não é uma defesa de acoplamento alto. É uma defesa de acoplamento explícito, intencional e alinhado ao domínio.

O objetivo não deve ser reduzir dependências a zero. Isso é fantasia. O objetivo deve ser refletir as conexões reais do sistema de forma que as pessoas consigam entender impacto, mudança e responsabilidade.

## Hierarquia não é sinônimo de reutilização por superclasse

Herança é o pilar mais perigoso dessa "lista popular" quando vira mecanismo padrão de reutilização.

O problema da "classe-base frágil" é conhecido: uma mudança aparentemente segura em uma superclasse pode quebrar subclasses que dependiam, direta ou indiretamente, de detalhes da implementação herdada.

Isso não significa que herança nunca deve ser usada. Significa que herança de implementação cria uma forma forte de dependência, e essa dependência pode te custar muito caro naquele momento "its fine!" da manutenção.

Grady Booch formulou que, um dos elementos centrais do *object model* é hierarquia, não simplesmente herança de classe. Hierarquia é uma forma de organizar abstrações, incluindo relações como "é um tipo de" e "é parte de". Herança é apenas uma implementação possível, e nem sempre a melhor.

A recomendação "favor object composition over class inheritance", popularizada pelo livro *Design Patterns*, não nasceu como moda. Ela reconhece que composição tende a preservar melhor o encapsulamento e reduzir dependências de implementação. Ainda assim, composição também tem custo: sistemas baseados em muita delegação dinâmica podem ficar difíceis de seguir. A maturidade não é trocar um dogma por outro, mas escolher a relação que torna o modelo mais claro. 

## Quando o código encontra o modelo mental

Se os pilares clássicos não bastam para trazer clareza, para onde devemos olhar?

Uma resposta forte vem do DCI: Data, Context and Interaction. O DCI foi originado por Trygve Reenskaug, criador do MVC, e desenvolvido com contribuição central de James Coplien.

A ideia do DCI é separar três perspectivas:

- Data: os objetos que representam o que o sistema é.
- Context: o cenário ou caso de uso em que os objetos participam.
- Interaction: a colaboração entre papéis dentro daquele contexto.

Em OO tradicional, muita lógica de caso de uso acaba espalhada por várias classes porque tentamos encaixar comportamento dinâmico dentro da estrutura estática dos dados. O resultado é que o fluxo do negócio fica fragmentado.

No DCI, objetos de dados continuam existindo, mas eles assumem papéis dentro de um contexto. O contexto explicita quais objetos participam do caso de uso e quais responsabilidades eles assumem naquele momento.

Isso resolve parte do paradoxo da coesão. Em vez de criar classes gigantescas que tentam fazer tudo, ou classes anêmicas cercadas por serviços genéricos, o DCI busca expressar a narrativa do caso de uso em um lugar mais legível. O desenvolvedor consegue enxergar a interação completa com menos saltos mentais.

O ponto não é que DCI seja a única resposta correta. O ponto é que ele recupera algo que a lista dos quatro pilares não enfatiza: software é comportamento em contexto, não apenas estrutura em classes.

## Pensamento final

A orientação a objetos que Alan Kay defendia não era uma obsessão por classes, hierarquias ou getters e setters. Era uma visão de objetos autônomos, protegendo seu estado e colaborando por mensagens.

Abstrações precisam ser habitáveis e ligadas ao domínio e sistemas são definidos por interações, não por partes isoladas. 

O meu conselho mais honesto para você seria: "evite a lista dos quatro pilares como um mantra e busque entender o que realmente torna um sistema orientado a objetos".

Então, da próxima vez que você desenhar um sistema, não pergunte apenas se aplicou encapsulamento, herança, abstração e polimorfismo.

**Pergunte também:**

- Quais mensagens existem entre as partes?
- O domínio está explícito ou escondido atrás de nomes genéricos?
- As dependências reais estão visíveis?
- A hierarquia ajuda ou prende?
- O caso de uso pode ser lido como uma interação compreensível?

No fim, Software é um meio de comunicação entre pessoas sobre um domínio em movimento.
